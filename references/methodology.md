# 起号分析方法论 & 技术细节

## 一、为什么这套数据能还原「起号过程」

X 不公开历史粉丝曲线,但**每条推文的发布时间 + 互动数(赞/转/评/收藏/浏览)是公开的**。
把一个账号的全部推文按时间排开,就能逆向出它的成长轨迹:

| 你想知道的 | 用什么数据代理 |
|---|---|
| 它什么时候真正开始做? | 月度活动量的突变点(起号拐点) |
| 注册后是不是先养号? | 注册时间 → 第一个高产月之间的「休眠期」 |
| 涨粉靠什么内容? | 内容支柱(高频词)+ 爆款帖的共同特征 |
| 涨粉是否在加速? | 原创均赞/均浏览随月份的斜率(增长代理) |
| 早期怎么冷启动的? | 回复:原创比、蹲守了谁的评论区 |
| 标题怎么写的? | Top 高赞帖首行的钩子句式分布 |

## 二、起号四阶段框架(读报告时套这个)

1. **占位期**:注册但不发,或零星试水。很多大V账号名是后改的。
2. **起号拐点**:某月突然放量(原创+回复同时飙升)。看 `inflection_month`。
   - 高频铺量 + 重度回复蹲守 是最常见的冷启动组合拳。
3. **找到爆款公式**:某几条帖子互动远超均值后,后续内容向其收敛(选题/格式/钩子趋同)。
4. **稳定增长**:均赞抬升、互动率稳定,开始沉淀付费/转化入口(简介里的咨询/产品链接)。

## 三、关键指标怎么解读

- **回复 : 原创比**:>2 通常意味着「评论区蹲守」是主要冷启动手段(在头部账号下高质量回复,蹭曝光导流)。看 `top_reply_targets` 就是它蹲守的对象。
- **自我续写(self_replies)**:回复对象=自己,即发 thread/长串,内容型创作者的标志。
- **收藏率(bookmark_rate)**:比赞更能反映「干货价值」——高收藏=工具教程类实用内容。
- **互动率(likes/views)**:稳定在 0.3%~0.5% 属健康;骤降可能是买量或蹭了泛流量话题。
- **长推占比**:>280 字的长图文多,说明走「深度干货」而非「短平快段子」路线。
- **均赞斜率**:`growth_trend.slope_per_month`,正且大=账号还在上升期。

## 四、技术细节与已知坑

### 4.1 突破 3200 条上限(核心)
- X 的**用户时间线接口(UserTweets)有约 3200 条的回溯上限**,大号拿不到早期帖。
- 但 **`from:handle since:YYYY-MM-DD until:YYYY-MM-DD` 搜索接口不受此限**:按月开窗逐段拉,理论可回溯到账号第一条。
- `acquire.py` 据账号总推文数自动选策略(<3000 走时间线,>3000 走按月搜索),也可 `--force-search` 强制。
- ⚠️ 搜索接口对**低互动的老推/纯回复**召回可能不全;时间线接口对小号反而最全。所以中小号优先时间线。

### 4.2 登录态:从运行中的 Chrome 抠 cookie(零配置的关键)
- twscrape 需要登录态(auth_token + ct0)。auth_token 是 httpOnly,`document.cookie` 读不到。
- 解法:连 Chrome 的 **浏览器级调试端点** `ws://127.0.0.1:<port>/devtools/browser`,调 `Storage.getCookies`,**httpOnly 也能拿**。见 `scripts/get_cookies.mjs`。
- 前提:Chrome 用 `--remote-debugging-port=9222` 启动且已登录 x.com。
- 这比「让用户去开发者工具手抄 token」体验好得多,是本 skill 的差异化卖点。
- ⚠️ **独占限制**:Chrome 的 `/devtools/browser` 端点同一时刻只接受一个客户端。若已有别的 CDP 工具(如某些常驻 inspector/proxy)占着调试端口,`get_cookies.mjs` 会超时。解决:关掉那个工具后重试,或直接传 `--cookies` / 设 `X_COOKIES`(脚本也会缓存到 `/tmp/xcookies.json` 复用)。

### 4.3 代理
- 国内直连 x.com 不通。`acquire.py` 自动复用 `HTTPS_PROXY` 或探测本地 7890(ClashX/Mihomo)。
- twscrape 默认不吃 env 代理,必须显式 `API(proxy=...)` 和 `add_account(proxy=...)`,脚本已处理。

### 4.4 作者过滤(数据洁癖)
- `user_tweets_and_replies` 会把「该账号回复过的别人原帖」(对话上下文)也带回来,这些是别人的高赞推,不过滤会污染「爆款 Top」。
- 脚本按 `author_id == 目标uid` 过滤,只留本人的推。

### 4.5 限流
- 单账号会被 X 限流,大号采集中途会出现 `No account available, next available at ...`,twscrape 自动等待。耐心等,别中断。
- 想快:给 twscrape 加多个账号(多 cookie 轮换)。

### 4.6 ⚠️ 用「采集小号」,别用主号
- twscrape 是**拿你登录 Chrome 的那个账号的身份**去调 X 内部接口的。那一点点"非官方用法"的风控风险落在**这个号**头上(不是被分析的对象)。
- **强烈建议**:专门养一个 X 小号,只用它登录带调试端口的那台 Chrome,采集走小号 cookie,主号永不沾风险。
- 风险主要来自「高频猛刷大号」。偶尔抓个几千条的号基本无碍;批量/反复全量采集才危险。

## 五、Python 版本
- twscrape 要 **3.10+**。macOS 系统自带常是 3.9——用 `brew install python@3.11` 或已有的 `python3.11`,在产出目录建 venv,别污染系统环境。
